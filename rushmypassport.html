Overview

From inconsistency to a single source of truth

Rush My Passport is a US-based service that expedites passport and travel document applications. When I joined, the product team had grown rapidly — three designers, four engineers — with no shared component library, no naming conventions, and no process for design changes to reach production consistently.

I was brought in to fix that: to design and document a system that would let the team move faster without breaking consistency, and would scale as the team continued to grow.

Scope

Full token architecture (primitive → semantic → component), 48 documented components, contribution guidelines, and a governance process for system evolution.

01 — Audit

Mapping the inconsistency

The first step was an interface audit: I cataloged every unique color value, type style, spacing value, and component variant in production. The result was sobering — 47 unique color values where there should have been 12, and 23 different button styles.

02 — Token Architecture

Primitive → Semantic → Component

I designed a three-tier token system. Primitive tokens are the raw values (colors as hex, spacing as rem). Semantic tokens assign meaning (color/background/primary, space/component/padding-sm). Component tokens bind semantics to specific elements (button/background/default).

This architecture means a brand rebrand touches only the primitive tier. A UX pattern change touches only the semantic tier. Component internals change only at the component tier — nothing cascades unexpectedly.

03 — Component Library

48 components. Every state documented.

I built the component library in Figma with auto-layout, component properties, and variant coverage for every interactive state: default, hover, focused, active, disabled, loading, and error. Engineers received exact token names alongside every spec.

04 — Documentation & Governance

A system only works if people actually use it

I wrote usage guidelines for every component: when to use it, when not to, which props are required vs. optional, and accessibility requirements. For governance, I created a contribution process — a clear path for proposing new components, with criteria for "add to system" vs. "one-off".
